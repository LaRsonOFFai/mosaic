# MOSAIC Product Requirements Document

Status: Draft 0.1

## 1. Purpose

MOSAIC is a research prototype of an encrypted layer-3 tunnel. A client reads IPv4 and IPv6 packets from a packet device, transfers them through a secure session over a pluggable carrier, and a server writes the reconstructed packets to its packet device or a controlled test sink.

The distinguishing research component is a scheduler that can fragment, coalesce, delay, pad, and interleave encrypted records according to an explicit cover profile. The scheduler is evaluated against ordinary traffic generated by an operator-controlled decoy application.

## 2. Product goals

### G-001 — Correct IP tunneling

Transfer IPv4 and IPv6 packets end to end without modifying their contents, subject to documented MTU and packet-size limits.

### G-002 — Transport independence

The protocol core must work over an in-memory carrier, a localhost test carrier, and an HTTP/3 carrier without changing packet, codec, reassembly, or session semantics.

### G-003 — Safe parsing

Arbitrary input must not cause panics, unbounded allocation, integer overflow, endless loops, or uncontrolled queue growth.

### G-004 — Standard cryptography

Use a standard authenticated key-establishment construction and standard AEAD. No custom encryption or home-grown key exchange.

### G-005 — Bounded packet pipeline

Fragmentation, reassembly, queues, timeouts, retry behavior, and padding overhead must have explicit configurable bounds.

### G-006 — Measured classification resistance

Traffic-shaping claims must be expressed as measured comparisons with baselines. The project must never promise that traffic is undetectable.

### G-007 — Recoverable system integration

TUN and routing changes must be explicit, reversible, and isolated from protocol logic.

## 3. Non-goals

MOSAIC does not aim to provide:

- anonymity against a global observer;
- protection after client or server compromise;
- guaranteed access through every network policy;
- evasion of server-IP or domain blocking;
- third-party domain fronting;
- impersonation of commercial services;
- hidden installation, persistence, or privilege escalation;
- covert use of other people's infrastructure;
- a production security guarantee before independent review.

## 4. Supported environment

Initial target:

- Linux client and server;
- stable Rust;
- IPv4 and IPv6 inner packets;
- local/in-memory tests on all development platforms supported by Rust;
- privileged TUN tests only on Linux and only when explicitly requested;
- HTTP/3 tests only against an operator-controlled endpoint.

Windows, macOS, Android, iOS, multipath, roaming, and mobile power optimization are future work.

## 5. Functional requirements

### FR-001 — Packet-device abstraction

The core consumes and produces opaque IP packets through an abstraction independent of Linux TUN. An in-memory implementation is mandatory for tests.

### FR-002 — Packet validation

Before accepting an inner packet, validate at minimum:

- non-empty input;
- IP version is 4 or 6;
- minimum header length;
- declared packet length is consistent with received length when the protocol provides such a field;
- packet size does not exceed configured limits.

The first prototype need not validate transport-layer checksums.

### FR-003 — Fragmentation

Packets larger than the configured record payload budget are split into fragments. Each fragment identifies:

- packet identifier;
- total packet length;
- byte offset;
- fragment payload length;
- first/last semantics or an equivalent canonical representation.

### FR-004 — Reassembly

Reassembly must:

- support out-of-order fragments;
- accept exact duplicate fragments only when bytes are identical;
- reject overlapping fragments with conflicting bytes;
- reject inconsistent total lengths;
- enforce per-packet and global memory limits;
- enforce a maximum number of incomplete packets;
- expire incomplete packets;
- emit a packet exactly once.

### FR-005 — Cell codec

Cells use a canonical versioned binary encoding. The streaming decoder supports:

- one cell split across arbitrary input chunks;
- several cells in one input chunk;
- truncated input;
- malformed input;
- bounded buffering;
- clean recovery policy defined by the specification.

### FR-006 — Session establishment

The client authenticates the server using configured key material. Client authorization uses a standard authenticated mechanism selected by ADR. The handshake derives independent send and receive keys.

### FR-007 — Record protection

After the handshake, records use authenticated encryption. Nonces must be unique per key and direction. Header fields required before decryption must be included in authenticated associated data.

### FR-008 — Replay and epoch handling

The session defines packet numbers, anti-replay behavior, key epochs, and key-update rules. Duplicate protected records must not be delivered twice.

### FR-009 — Carrier abstraction

A carrier transfers opaque protocol messages or byte streams. The abstraction exposes:

- reliable ordered delivery capability;
- optional datagram capability;
- maximum safe message size;
- connection lifecycle;
- backpressure and errors.

The protocol cannot assume that one carrier write equals one carrier read.

### FR-010 — Local carrier

A loopback-only test carrier provides deterministic end-to-end testing without TUN or external networking.

### FR-011 — Linux TUN adapter

A separate Linux crate reads and writes raw IP packets from a TUN interface. Privileged execution is opt-in. Automated default tests do not require TUN.

### FR-012 — Route safety

If route management is implemented, it must:

- require explicit user flags;
- protect the route to the server endpoint;
- record each applied change;
- roll changes back on normal shutdown;
- provide documented manual recovery;
- refuse ambiguous or unsafe configuration.

### FR-013 — HTTP/3 carrier

The HTTP/3 carrier uses a maintained standards-compliant stack and valid TLS certificate verification. It connects only to an operator-controlled service.

### FR-014 — Decoy application

The HTTP/3 server exposes an ordinary harmless application independent of the tunnel. Unauthenticated requests receive ordinary application behavior rather than proxy-specific or tunnel-specific errors.

### FR-015 — Scheduler

The scheduler operates between encrypted records and carrier events. It supports:

- disabled/fast mode;
- bounded balanced mode;
- strict profile mode;
- fragmentation and coalescing of encrypted record bytes;
- interleaving between multiple flows or batches;
- padding with explicit byte and delay budgets;
- deterministic seeded simulation;
- observable metrics.

### FR-016 — Cover profiles

A cover profile is versioned data defining permitted event sizes, delays, directions, burst lengths, state transitions, and maximum queue delay. Invalid profiles are rejected with bounded parsing.

### FR-017 — Configuration

Client and server configuration is explicit, validated, and printable with secrets redacted. Unknown fields follow a documented compatibility policy.

### FR-018 — Metrics

Expose at minimum:

- inner bytes and packets;
- carrier bytes and events;
- padding bytes;
- dropped packets by reason;
- queue occupancy high-water marks;
- reassembly expirations;
- handshake and reconnect outcomes;
- latency summary without logging payload contents.

### FR-019 — Graceful shutdown

Client and server stop accepting new work, drain within a configured deadline, close the session, close packet devices, and attempt route rollback.

### FR-020 — Error handling

Errors are categorized as configuration, authentication, protocol, resource-limit, carrier, packet-device, or internal invariant failures. User-facing messages must not leak secrets.

## 6. Non-functional requirements

### NFR-001 — Memory safety

No unsafe Rust without explicit justification. Input-derived allocation is limited and checked before allocation.

### NFR-002 — Determinism

Core tests inject clocks and deterministic randomness. No sleep-based correctness test when a simulated clock can be used.

### NFR-003 — Performance

The fast mode should avoid unnecessary packet copies where safely practical. Performance optimization must not weaken validation.

### NFR-004 — Maintainability

Wire-format, cryptographic, and privilege-boundary changes require ADRs and specification updates.

### NFR-005 — Observability

Structured logs are supported. Secrets, raw packet payloads, keys, authentication proofs, and full tokens are never logged.

### NFR-006 — Reproducibility

Evaluation tools save configuration, random seed, source revision, and summary metrics with each result.

### NFR-007 — Compatibility

Version negotiation and feature negotiation must fail closed for unknown mandatory features and remain extensible for optional features.

## 7. Initial configurable limits

Exact defaults may be adjusted by ADR, but every implementation must expose and validate limits equivalent to:

- maximum inner packet size: 65,535 bytes;
- maximum encrypted record plaintext: 16 KiB;
- maximum encoded cell body: 64 KiB;
- maximum incomplete reassemblies: finite and configurable;
- maximum total reassembly bytes: finite and configurable;
- maximum per-direction queued bytes: finite and configurable;
- maximum handshake message: finite and configurable;
- maximum cover-profile size and state count: finite and configurable;
- maximum shutdown drain duration: finite and configurable.

## 8. Acceptance summary

The prototype is accepted only when:

1. arbitrary codec and reassembly input is bounded and fuzz-tested;
2. a synthetic IPv4 and IPv6 packet completes an encrypted localhost round trip;
3. Linux TUN integration works in an isolated namespace without changing the host default route;
4. HTTP/3 integration works against an operator-controlled endpoint with certificate validation enabled;
5. malformed and unauthenticated requests do not expose tunnel-specific behavior at the decoy endpoint;
6. scheduler modes produce documented overhead and delay metrics;
7. evaluation reports false-positive and true-positive measurements rather than marketing claims;
8. all mandatory formatting, linting, unit, integration, and negative tests pass.
